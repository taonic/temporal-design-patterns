# runtime/

Per-language toolchain assets shared by every pattern in [`../patterns/`](../patterns/).

Each pattern (`continue-as-new`, `saga`, …) only ships its own workflow
source; the SDK dependency files, build configuration, and any compile-cache
warmup live here so the Daytona snapshot is keyed on language, not on
`(pattern, language)`. One cached image per language serves every pattern
that uses it.

## Layout

```
runtime/
├── typescript/
│   ├── package.json          # @temporalio/* dependencies
│   ├── tsconfig.json
│   └── package-lock.json     # checked in for reproducible installs
├── python/
│   └── pyproject.toml        # temporalio dependency, managed by uv
├── go/
│   ├── go.mod                # go.temporal.io/sdk
│   ├── go.sum                # generated by `go mod tidy`, must be committed
│   └── warmup.go             # SDK-import stub compiled at image-build time
└── java/
    ├── pom.xml               # io.temporal:temporal-sdk + slf4j-simple
    └── _warmup/Warmup.java   # SDK-reference stub compiled at image-build time
```

## Local runs from a pattern dir

Each pattern's per-language source dir (e.g. [`../patterns/saga/go/`](../patterns/saga/go/))
holds **symlinks** back to this directory for every toolchain file it needs:

| Pattern dir contains symlinks for |
| ---: |
| TypeScript: `package.json`, `package-lock.json`, `tsconfig.json`, `node_modules/` |
| Python: `pyproject.toml` |
| Go: `go.mod`, `go.sum` |
| Java: `pom.xml` |

So you can run a sample directly from its source dir:

```bash
cd sandbox-runner/patterns/saga/go && go run worker.go workflows.go activities.go shared.go
cd sandbox-runner/patterns/saga/typescript && npx tsx worker.ts
```

The samples assume a Temporal dev server on `localhost:7233` (start one with
`temporal server start-dev`).

The manager itself reads from this directory directly, so the symlinks are
purely a developer-ergonomics convention. To wire a new pattern, recreate
the symlinks (one `ln -s` per file) — there's no script for it because the
set is small and runs once per pattern.

## How the manager uses these files

[`../src/manager.ts`](../src/manager.ts) defines a zero-arg `Image` factory
per language. Each factory:

1. Picks a base image (`node:20-bookworm-slim`, `ghcr.io/astral-sh/uv:python3.12-bookworm-slim`, `golang:1.22-bookworm`, `eclipse-temurin:17-jdk`).
2. Installs the Temporal CLI.
3. `addLocalFile`s the deps file (and warmup, if applicable) from this directory.
4. Runs the language's install/cache-warm step.

The fingerprint Daytona uses to key the cached snapshot includes only the
files this directory contributes, so changes to a pattern's workflow source
never invalidate the snapshot.

## Bumping SDK versions

| Language | Edit | Then |
| --- | --- | --- |
| TypeScript | `typescript/package.json` | `cd runtime/typescript && npm install` to refresh the lockfile |
| Python | `python/pyproject.toml` | `uv` regenerates the lock inside the sandbox; nothing to commit locally |
| Go | `go/go.mod` | `cd runtime/go && go mod tidy` to refresh `go.sum` (must be committed) |
| Java | `java/pom.xml` | nothing — Maven resolves on next image build |

After any of these, the next launch in that language rebuilds the snapshot
once.

## Why warmup files exist

Compiled languages (Go, Java) would otherwise spend the first runtime
`go run` / `mvn compile` invocation compiling the SDK itself, racing the
manager's worker-startup window. The warmup files are tiny stubs that
import every SDK package the workflow code uses; the image factory compiles
them at bake time so the SDK lands in the build cache, then deletes the
sources from the snapshot. Subsequent runtime invocations only have to
compile the user's small main package.

**Gotcha (Go):** the warmup filename must **not** start with `_`. Go's
package loader filters `_*.go` files even when listed explicitly on the
build command line, surfacing as `package command-line-arguments: no Go
files in /opt/app`.

## Adding a new language

1. Create `runtime/<language>/` with at minimum a deps file. Add a warmup
   stub if the language has a compile step.
2. Add a factory entry in `IMAGE_FACTORIES` in [`../src/manager.ts`](../src/manager.ts).
3. Add a `DAYTONA_LANGUAGE_HINT` mapping (Daytona's enum may not include the
   new language; pick the closest accepted value).
4. Wire CodeMirror syntax via a `@codemirror/lang-*` package and a branch in
   `languageExtension()` in [`../../docs/.vitepress/theme/components/DaytonaRunner.vue`](../../docs/.vitepress/theme/components/DaytonaRunner.vue).
5. Add a language entry to each pattern's `pattern.json` (or to whichever
   patterns you want to support the language for).
